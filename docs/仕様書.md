## 1. プロダクト概要

### 1.1 目的

テクノ中部社長が経営課題を入力すると、複数の専門AIエージェントが3ラウンドの熟議を行い、「第三の解」を提示する経営参謀AIシステム。

日立・ハピネスプラネット社の FIRA（Happiness Planet FIRA）と同等の機能を、テクノ中部の経営文脈に特化して自社構築したもの。プロダクト名「ARIF」は FIRA の逆順であり、「FIRAを超える・見抜く」という意志を示す。

### 1.2 プロダクト名の意味

| 観点 | 内容 |
|------|------|
| 表記 | ARIF（アリフ） |
| 由来① | FIRA のアナグラム（逆順） |
| 由来② | アラビア語・ペルシャ語で「知者・賢人・物事の本質を知る者」 |
| 略称展開 | **A**utonomous **R**easoning **I**ntelligence **F**ramework |

### 1.3 理論的背景：認知枠干渉モデル（CFIM）

ARIF の設計思想は「認知枠干渉モデル（Cognitive Frame Interference Model）」に基づく。

> **「意味的に最も遠い認知フレームを意図的に衝突させると、どちらにも含まれていなかった意味（第三の解）が生まれる」**

これは波の干渉現象と同じ構造であり、FIRAが借用した量子代数知能とは異なり、完全に透明・再現可能な手法である。

## 3. 機能要件

### 3.1 熟議機能（コア）

| 要件ID | 要件 |
|--------|------|
| F-001 | ユーザーが経営課題をテキストで入力できること |
| F-002 | 5体の専門エージェントが Round 1 で独立回答できること（相互参照なし） |
| F-003 | Round 1 の全意見を共有した後、Round 2 で各エージェントが反論・深掘りできること |
| F-004 | Round 2 では全エージェントに「他者の意見の最も危険な前提への反論」が義務付けられること |
| F-005 | Round 3 で統合プロンプトが「第三の解」を起草できること |
| F-006 | Round 3 の後、Pre-mortem（事前検死）として「この判断が失敗する場合の原因5つ」を自動生成できること |
| F-007 | 熟議結果をユーザーに提示できること |
| F-008 | 熟議の進行状況をリアルタイムで表示できること（「○○エージェントが発言中…」） |

### 3.2 記憶機能

| 要件ID | 要件 |
|--------|------|
| M-001 | 過去の議論セッションを Supabase に永続化できること |
| M-002 | 直近 N 件の議論を次回セッションに文脈注入できること |
| M-003 | 会社基本情報（事業概要・強み・懸案事項等）を管理できること |
| M-004 | 会社情報・過去議論を pgvector でベクトル検索できること（RAG） |
| M-005 | セッション横断の洞察（決定事項・懸念・パターン）を蓄積できること |

### 3.3 フィードバック機能

| 要件ID | 要件 |
|--------|------|
| B-001 | 社長がセッションに 1〜5 の総合評価を付けられること |
| B-002 | 「最も参考になったエージェント」「最も不要だったエージェント」を記録できること |
| B-003 | 実際にどう意思決定したかを後日入力できること（改善ループのため） |

### 3.4 API

| 要件ID | 要件 |
|--------|------|
| A-001 | POST /deliberations で熟議を非同期実行できること |
| A-002 | GET /deliberations で過去の議論一覧を取得できること |
| A-003 | GET /deliberations/{id} で特定の議論詳細を取得できること |
| A-004 | GET /deliberations/{id}/stream でSSEによる進行状況をリアルタイム配信できること |

## 4. エージェント構成

### 4.1 確定構成：5体 ＋ 統合プロンプト

Opus顧問の審査により、初期フェーズは **5体** に絞る。

| # | エージェントID | 役割 | 視点 | モデル | 温度 |
|---|------------|------|------|--------|------|
| 1 | strategist | 経営戦略家 | 5〜10年の事業方向性・競合・ポジショニング | Opus | 0.7 |
| 2 | cfo | 財務・投資家 | ROI・キャッシュフロー・投資回収・数字 | Opus | 0.3 |
| 3 | engineer | 技術・現場 | 実現可能性・現場の実態・技術リスク | Opus | 0.4 |
| 4 | market | 顧客・市場 | 顧客視点・競合分析・市場動向 | Opus | 0.8 |
| 5 | risk | リスク・法務 ＋ 反論役 | 最悪シナリオ・規制・反論（悪魔の代弁者を統合） | Opus | 0.2 |
| — | synthesizer | 統合者 | 第三の解の起草（Round 3 専用プロンプト） | Opus | 0.5 |
| ※ | philosopher | 歴史・哲学 | 本質を問う・長期視点（任意・動的ON/OFF） | Opus | 1.0 |

> **「悪魔の代弁者」は独立エージェントとしては設けない。**  
> 代わりに Round 2 の全エージェントのプロンプトに「他者意見への反論義務」を構造的に埋め込む。

### 4.2 Round 2 の反論義務テンプレート

```
【必須タスク】
1. 他のエージェントの意見で「最も危険な前提」を1つ特定し、反論せよ
2. 自分の Round 1 意見で「最も弱い部分」を1つ自己批判せよ
3. 修正した自分の立場を述べよ

※ 全員一致が生じた場合は、それ自体がリスクシグナルである。
  なぜ全員が同じ結論に至ったのか、構造的原因を分析せよ。
```

### 4.3 「第三の解」の品質基準

統合プロンプトは以下の基準を全て満たす出力を求める。

| # | 基準 | 内容 |
|---|------|------|
| 1 | 新規性 | Round 1 で誰も提案していない要素を含む |
| 2 | 統合性 | 少なくとも 3 つのエージェントの論点を取り込む |
| 3 | 実行可能性 | 具体的な次のアクションが明記されている |
| 4 | リスク認識 | 主要リスクとその緩和策が含まれている |
| 5 | 時間軸 | 短期（3ヶ月）と中期（1〜3年）のアクションを区別する |

## 5. 非機能要件

| 項目 | 要件 |
|------|------|
| 応答時間 | 1回の熟議を 3分以内に完了（SSE で進行状況を随時表示） |
| コスト上限 | 1回の熟議を $0.20（約¥30）以内に抑える |
| 可用性 | 平日業務時間中（9:00〜18:00）に利用可能なこと |
| ログ保存 | 全熟議のログ・コスト・レイテンシを永続化すること |

## 6. セキュリティ要件

| 要件ID | 要件 |
|--------|------|
| S-001 | APIキー・Supabase資格情報を環境変数で管理し、コードに直書きしないこと |
| S-002 | モデルに送信する入力から個人名・取引先名をマスキングできること（将来対応） |
| S-003 | Anthropic API 経由のデータがモデル学習に使用されないことを利用規約で確認済みであること |
| S-004 | セッション情報へのアクセスを認証済みユーザーのみに制限すること |
| S-005 | モデルバージョンを日付付きバージョンでピン留めし、意図せぬ差し替えを防ぐこと |

## 8. 開発ロードマップ

| Phase | 内容 | 期間目安 |
|-------|------|---------|
| **0** | Jupyter Notebook でエージェントペルソナ手動検証（プロンプト品質確定） | 1週間 |
| **1** | FastAPI + 熟議エンジン（CLI版・非同期並列） | 2週間 |
| **2** | Supabase スキーマ構築・セッション永続化 | 1週間 |
| **3** | SSE + HTMX フロントエンド（リアルタイム進行表示） | 1週間 |
| **4** | RAG（過去議論参照）+ フィードバックループ | 2週間 |
| **5** | Pre-mortem 自動化・意見多様性チェック | 1週間 |

> **Phase 0 が最重要。** プロンプトの質がシステム全体品質の 80% を決定する。  
> コードを書く前に手動で 3 ラウンドを回し、各エージェントの出力品質を検証すること。
